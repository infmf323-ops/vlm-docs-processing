# PaddleOCR-VL MRZ: разбор и восстановление пайплайна (31.05.2026)

Документ фиксирует находки сессии отладки: почему предыдущие эксперименты (v1–v4) были
невалидны, что именно починено, и какой получен первый честный бейзлайн. Пригодно для
главы «методология / угрозы валидности» тезиса.

## 1. Корневая причина: сломанный инференс модели

Эксперименты v1–v4 (pretrain + Russian finetune LoRA) обучались и оценивались на модели,
прямой проход которой был **математически испорчен**.

- На Kaggle стоял `transformers` 5.x, тогда как PaddleOCR-VL официально рассчитан на
  **`transformers == 4.55.0`** (источник: карточка модели PaddlePaddle/PaddleOCR-VL).
- Чтобы модель «завелась» под 5.x, ранее были добавлены три монкипатча. Из них:
  - **patch2** (шим сигнатуры `create_causal_mask`: `inputs_embeds` → `input_embeds`) —
    легитимен и необходим.
  - **patch1 и patch3** (подмена RoPE на «default») — **вредны**: они затирали родной
    multimodal-RoPE (mrope) модели, ломая позиционное кодирование. Итог — модель на
    свободной генерации выдавала мусор, хотя teacher-forcing loss падал.
- Следствие: снижение val-loss в v1–v4 не отражало реального качества; eval-метрики
  (char_accuracy ~0.03) были следствием сломанного форварда, а не данных/обучения.

**Исправление.** `transformers==4.55.0` + загрузка `AutoModelForCausalLM(trust_remote_code=True,
torch_dtype=float16/bfloat16)` + **только patch2**, без RoPE-патчей. После этого база
начинает корректно читать текст паспортов.

## 2. Баг синтетического генератора MRZ

`backend/scripts/generate_synthetic_mrz_data.py` заявлял «valid TD3 with correct check digits»,
но по факту валидных MRZ было **29 из 350 (8%)**.

- Причина: композитная контрольная цифра считалась не по ICAO 9303 — в неё включался пол
  (`sex`) и поле `optional` бралось 15 символов без собственной контрольной цифры.
  По стандарту композит покрывает позиции line2 1–10, 14–20, 22–43 (пол и гражданство
  исключены), а optional = 14 символов + своя контрольная цифра.
- **Исправлено** (`build_mrz`): корректная раскладка + композит. Проверка: **500/500 валидных**.

Вывод для тезиса: обучающая синтетика v1–v4 содержала ~92% MRZ с неверными контрольными
суммами — нереалистичные данные, на которых нельзя строить ни обучение, ни пост-коррекцию.

## 3. Аудит данных

- Реальных паспортов с пригодным MRZ-эталоном в проекте **нет**: спесимены `ud_biometrics`
  имеют нестандартный/битый MRZ, у части нет меток вовсе; паспорт реального человека —
  это PII и в датасет не включается.
- Внешняя синтетика **Unidata (`ud-synthetic`)** — фотореалистична, многострановая, с метками
  MRZ; валидны по чек-суммам ~57%.
- Локальный MIDV — это MIDV-DM (классификация подлинности), без разметки MRZ.

## 4. Честный held-out эвал

- Собран `passport_eval_unidata` — **25 паспортов Unidata, 5 стран** (chinese, greek, indian,
  russian, turkish), по 5 на страну, только валидные по TD3 эталоны.
- Кропы MRZ выверены по-странам (вертикальные доли), каждый содержит обе строки.
- Зафиксирован список исключения этих изображений из обучения (`EVAL_HELDOUT_EXCLUDE.txt`)
  для предотвращения утечки train/test.
- Метрики: char_acc (обе строки), точность полей line2 (28 симв.), line1/line2 exact,
  доля предсказаний, проходящих контрольные суммы TD3 (`mrz_td3.py`).

## 5. Первый честный бейзлайн (база PaddleOCR-VL, без дообучения)

| метрика | значение |
|---|---|
| char_acc (обе строки) | 0.70 |
| line1 exact | 0.68 |
| line2 exact | 0.08 |
| TD3 valid (предсказание) | 0.08 |
| лучшие случаи (turkish) | char 1.00, TD3 valid |

**Интерпретация.** Контент модель читает хорошо (line1 exact 0.68; в удачных случаях поля
line2 = 100%, MRZ проходит контрольные суммы). Узкое место — **консистентность формата**:
при жадной генерации модель после line1 залипает на заполнителе `<` и не доходит до line2.
Когда переход к line2 происходит — строка читается идеально.

## 5a. Декодирование: greedy vs beam search

Проверена гипотеза, что недобор line2 — чисто проблема декодирования.

- **Greedy**: чисто читает line1 (line1_exact 0.68), но часто теряет line2 — после line1
  залипает на заполнителе `<` и не доходит до второй строки.
- **Beam search (num_beams=4)**: чаще вытаскивает line2 (поля line2 = 100% на прочитанных),
  но вводит новые дефекты — иногда склеивает обе строки без переноса, иногда `<`
  декодируется как `LT`.

Вывод: **ни greedy, ни beam не дают стабильно полный валидный TD3-MRZ (2×44)**. Модель читает
*содержимое* MRZ хорошо, но не удерживает жёсткий формат. Это прямой аргумент в пользу
LoRA-дообучения (зашить формат), а не дальнейшей возни с декодированием.

Замечание по эвалу: один из 5 кропов indian оказался пустым (фиксированная доля 0.83–0.93
для 2-страничного разворота попадает в поле, а не в MRZ) — баг эвала, не модели; чинится при
сборке harness'а сравнения. Скорер дополнен устойчивым разбором склеенных строк (`split_mrz`).

## 5b. Результат LoRA-дообучения (фаза 2)

Дообучение на корректном фундаменте (transformers 4.55 + только patch2, без rope-патчей):
лёгкий LoRA (rank 8, alpha 16, LR 1e-4, 3 эпохи) на **280 синтетических MRZ-кропах с
валидным TD3**. Train loss 0.41 → 0.069, val loss 0.117 → **0.077** (на сломанной модели
в v4 «лучший» val был 1.99 — несопоставимо).

Оценка base vs LoRA на **том же held-out Unidata** (25 кропов, 5 стран; синтетика обучения
не пересекается с Unidata — утечки нет):

| метрика | base | LoRA | Δ |
|---|---|---|---|
| char_acc (обе строки) | 0.695 | **0.994** | +0.30 |
| line2 поля (28 симв.) | 0.293 | **0.999** | +0.71 |
| line1 exact | 0.680 | **1.000** | +0.32 |
| line2 exact | 0.080 | **0.760** | +0.68 |
| TD3 valid (предсказание) | 0.080 | **0.760** | +0.68 |

Вывод: база читала *содержимое* MRZ, но не держала формат 2×44 (line2 exact / TD3 valid ≈ 0.08).
Лёгкий LoRA на валидной синтетике зашивает формат — доля предсказаний, проходящих контрольные
суммы TD3, выросла с **8% до 76%**, char_acc до **0.99**. chinese/greek/indian/russian —
почти все валидны.

### Ограничение домена синтетики и его устранение (v6)
В v5 turkish давал char 0.97, но line2 exact = 0 у всех 5: турецкие паспорта заполняют
опциональное поле line2 национальным номером, а генератор v5 ставил там заполнитель `<`.
Исправление: генератор v6 в ~50% сэмплов заполняет опциональное поле реалистичными
данными (с корректной чек-цифрой и пересчётом композита). Переобучение на v6:

| метрика | base | LoRA v5 | LoRA v6 |
|---|---|---|---|
| char_acc | 0.695 | 0.994 | **0.999** |
| line1 exact | 0.680 | 1.000 | **1.000** |
| line2 exact | 0.080 | 0.760 | **0.920** |
| TD3 valid | 0.080 | 0.760 | **0.920** |

TD3-valid по странам (v6): indian/russian/turkish = 1.0, chinese/greek = 0.8. turkish
полностью исправлен (0 → 1.0). Остаточные 2 промаха из 25 (chinese, greek) — единичная
ошибка OCR в одном символе line2, не проблема формата; устранимо пост-коррекцией по
контрольным суммам TD3 (future work).

## 5c. Аугментация, пост-коррекция и hard-сабсет (v7)

Дальнейшие улучшения:
- **Аугментация генератора**: вариативность шрифтов, фонов (parchment/white/blue/green/pink/grey),
  шум, блюр, JPEG-подобная мягкость, скос ±3°. Объём 600/150 (всё валидно по TD3).
- **TD3 пост-коррекция**: перебор OCR-перепутываний (O↔0, I↔1, S↔5, …) так, чтобы прошли
  контрольные суммы. На синтетическом тесте восстанавливает **98%** одиночных OCR-ошибок.
- **Hard-сабсет**: 20 паспортов «на столе» (french/japanese/german/usa), MRZ под скосом —
  стресс-тест «in-the-wild».

Финальное сравнение base vs LoRA v7 (greedy):

**MAIN Unidata (n=25):**

| метрика | base | v7 | v7 + пост-коррекция |
|---|---|---|---|
| char_acc | 0.695 | 0.998 | — |
| line1 exact | 0.680 | 1.000 | — |
| line2 exact | 0.080 | 0.880 | — |
| TD3 valid | 0.080 | 0.880 | **1.000** |

**HARD «на столе» (n=20):**

| метрика | base | v7 | v7 + пост-коррекция |
|---|---|---|---|
| char_acc | 0.530 | 0.520 | — |
| line2 exact | 0.000 | 0.250 | — |
| TD3 valid | 0.000 | 0.250 | **0.350** |

Выводы:
1. **Основная задача решена**: на реалистичном held-out связка «правильная версия + кроп MRZ +
   лёгкий LoRA на валидной синтетике + пост-коррекция по контрольным суммам» даёт **100%**
   валидных по TD3 MRZ (с 8% у базы), char 0.998.
2. **Граница подхода**: на «in-the-wild» (скос, мелкий MRZ, фон) база не читает (0% valid),
   v7+аугментация поднимают до 25%/35%, но `char_acc` почти не меняется (0.52) — узкое место
   здесь **локализация и выравнивание (деског)**, а не формат. Future work: детектор MRZ-зоны
   + перспективное выравнивание перед OCR.

## 6. Следующие шаги

1. **Beam search** (`num_beams=4`) — проверить, решает ли проблему перехода к line2 чисто
   декодированием. Если да — вклад работы смещается в сторону пайплайна (версия + кроп +
   декодирование + пост-коррекция по контрольным суммам), и дообучение может быть не нужно.
2. Если формат не стабилизируется декодированием — **LoRA-дообучение** на корректной
   (валидной по TD3) синтетике поверх рабочего фундамента, с целью зашить жёсткий формат 2×44.
3. Перегенерировать обучающую синтетику починенным генератором; держать Unidata-эвал строго
   held-out.
4. Опционально расширить эвал «сложным» сабсетом (паспорта «на столе», со скосом) для оценки
   устойчивости.
